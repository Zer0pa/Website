# PRD: Deterministic Recursive Production Engineering

## Status

- status: `active engineering overlay`
- owner: `orchestrator`
- date: `2026-04-03`
- predecessor: `CLAW/PRD_CANONICAL_ROUTES_TO_PRODUCING.md`

## Purpose

Move the system from "route proof and recovery hardening" to "Codex-supervised recursive production engineering" without letting the website drift into probabilistic design.

This PRD governs system quality, not just route output.

## Hard Truth

The system is ready for supervised recursive work with Codex as the interface.

The system is not yet ready for unattended infinite automation.

Why:

- deterministic geometry exists but is not yet fully expanded into semantic, code-quality, contrast, and responsive release gates
- homepage and flagship closure are accepted, but the generic proof routes are still in progress
- the quality stack has not yet been elevated to a first-class production gate

## Core Doctrine

### 1. Math First

Layout, typography, spacing, color, and route truth are treated as explicit constraints.

No design change is accepted because it "looks better" unless it can be defended as one of:

- a declared geometry law
- a route-specific exception
- a contrast or legibility correction
- a semantic or search-friendly structure improvement
- a truth-preserving data integration improvement

### 2. Quality Is A Gate, Not Polish

The system must reject outputs that are visually close but structurally weak.

That means the following are production gates, not optional cleanup:

- semantic hierarchy
- code quality and lint discipline
- metadata and search discoverability
- responsive integrity
- contrast and legibility

### 3. Responsive Enters In Two Tiers

Responsive work is not a late surprise, but it is not all enforced at the same time.

- tier A from the beginning:
  - no horizontal overflow
  - no shell collapse
  - no broken action surfaces
- tier B after canonical parity:
  - full breakpoint matrix
  - typographic and rhythm consistency
  - large-screen and television behavior

### 4. Augmentation Comes After Truth And Form

No commercial wedge, interactive visual, or licensing flourish is allowed to outrun:

- truthful packet binding
- canonical geometry
- responsive integrity
- legibility and code quality

## Layered Work Program

### D0. Always-On Invariants

These are always active during every phase:

- build must pass
- parser falsification must pass
- path scope must be respected
- geometry must stay deterministic
- truth surfaces may not be fabricated
- critical or major regressions are automatic rejection

### D1. Structural Quality Foundation

Objective:

- make semantic quality, code quality, and search foundations measurable

Deliverables:

- lint configuration
- semantic and metadata audit
- route-level quality report
- production rule that quality findings are first-class blockers

Exit criteria:

- lint is configured and runnable
- a quality audit report exists for `/`, `/imc`, `/work/xr`, and `/work/ft`
- missing structural issues are visible as machine-readable evidence

### D2. Canonical Route Proof Continuation

Objective:

- continue canonical route proof without abandoning the system program

Scope:

- `/work/xr`
- `/work/ft`

Exit criteria:

- XR accepted under kernel law
- FT accepted as replication proof
- both routes replay through integration

### D3. Responsive Matrix Expansion

Objective:

- move from shell survival to true breakpoint proof

Breakpoints:

- laptop
- desktop-wide
- tablet
- mobile
- television

Exit criteria:

- no overflow across the matrix
- no collapsed hierarchy
- no unreadable type scaling
- large-screen behavior is explicit rather than incidental

### D4. Contrast And Legibility Refinement

Objective:

- eliminate low-contrast and near-invisible text decisions without changing the visual system irresponsibly

Exit criteria:

- contrast report exists
- failing contrast pairs are corrected or explicitly waived
- micro-copy, labels, off-black text, and off-white text remain legible against their true backgrounds

### D5. Layout Optimization After Parity

Objective:

- improve layout quality only after canonical parity is stable

Rules:

- no optimization that breaks the measured mockup basis
- no broad restyling
- only explicit improvements to spacing, alignment, density, and hierarchy

Exit criteria:

- improvements are explainable as laws or declared exceptions
- no regression to canonical reference parity

### D6. Augmentation And Data Surface

Objective:

- enrich the truthful product pages after form and truth are stable

Examples:

- license surfacing
- GitHub-connected status surfaces
- repo and primitive visualizations
- product-family augmentation rules by repo class

Exit criteria:

- augmentation backlog is seeded
- augmentations are subordinate to truth and geometry
- no augmentation introduces false claims or visual clutter

### D7. Codex-Supervised Recursive Operation

Objective:

- prove the system can continue improving with Codex as the operator interface

Requirements:

- D1 through D4 evidence exists
- D2 is complete through XR and FT
- recovery hardening remains current
- page squads and acceptance loops are explicit

Exit criteria:

- Codex can supervise bounded cycles without architectural ambiguity
- the system can improve recursively without deterioration
- the truthful claim becomes:
  - supervised recursive production is ready
  - unattended automation is the next step, not a fantasy claim

## Page Squad Model

Every important route family gets a bounded squad.

Shared role set:

- geometry architect
- frontend implementer
- code-quality auditor
- semantic and SEO auditor
- responsive architect
- contrast auditor
- truth auditor
- falsifier

The squad works in this order:

1. formulate
2. plan
3. implement
4. falsify
5. integrate
6. checkpoint

## Press-Go Rules

`press_go` stays `false` until:

- D1 has measurable evidence
- D2 completes XR and FT
- D3 and D4 have at least one clean proof bundle
- D6 is seeded but constrained
- recovery evidence remains current

## Non-Negotiables

- no creative drift
- no semantic sloppiness hidden by visual polish
- no duplicated route metadata left invisible
- no responsive breakage deferred indefinitely
- no low-contrast text accepted because it is fashionable
- no augmentation before truthful structure exists
- no claim of unattended autonomy before these layers are actually in place
