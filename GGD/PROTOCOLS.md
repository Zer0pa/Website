# GGD PROTOCOLS

These are the protocol families the deterministic website system should use.

## 1. map-existing-design

Purpose:

- map mockup geometry, live geometry, route truth, semantics, metadata, and code surfaces

When to use:

- before new autonomy
- when route truth is stale
- after major kernel changes

## 2. lock-conventions

Purpose:

- update or validate the geometry convention lock without ambiguity

When to use:

- before planning
- after introducing a new route family or token family

## 3. discuss-phase

Purpose:

- clarify gray areas before planning a new route or subsystem phase

Typical gray areas:

- flagship-only exceptions
- kernel vs route ownership
- breakpoint behavior
- truth-source priority
- acceptable augmentation scope

## 4. plan-phase

Purpose:

- convert one phase into bounded plans and waves

Planning rule:

- one plan should produce one verifiable change set
- one wave contains independent plans only

## 5. execute-phase

Purpose:

- run one wave at a time with lane-local ownership and evidence capture

Execution rule:

- implementation lanes change code
- verifier lanes do not bless anything without evidence

## 6. verify-work

Purpose:

- run named verification bundles and return severity-graded findings plus explicit gap output

Verification rule:

- no promotion on screenshots alone
- no promotion on operator intuition alone

## 7. debug-geometry

Purpose:

- isolate the first law or component boundary where route geometry diverges

Debug rule:

- start with spacing, grid, role mapping, and packet truth before stylistic speculation

## 8. replicate-family

Purpose:

- prove that a shared fix generalizes from one product page to another

Replication rule:

- reject any fix that only works for one repo-backed page

## 9. augmentation-review

Purpose:

- evaluate enhancements only after canonical closure

Augmentation rule:

- enhancement must not corrupt route truth, semantic clarity, or geometry law
